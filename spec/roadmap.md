# Roadmap — Auto-Podcaster

## What This Agent Does

Auto-Podcaster turns a one-line topic into a natural, multi-host podcast that streams to the
browser in real time, with a downloadable audio file when finished. The user types a topic,
picks a fixed cast of 2–3 AI host personas, and presses Generate. The backend writes a coherent
conversation turn-by-turn (Gemini), synthesizes each line to speech (edge-tts), and streams the
audio as it is produced. One-shot per run: no mid-stream control, no cross-session memory for v1.

## Who Uses It

A single user (the repo owner) who wants a quick, listenable audio discussion on any topic without
recording or editing. No multi-tenant auth, no accounts.

## Core Problem Being Solved

Generating a polished multi-host audio conversation normally needs a human, a script, and a TTS
toolchain. Auto-Podcaster collapses that into one text box + a Generate click.

## Success Criteria

- [ ] A user enters a topic + selects 2 hosts and hears a coherent two-person conversation stream live in the browser.
- [ ] The same run yields a downloadable audio file (mp3) when the conversation ends.
- [ ] The dialogue is generated by the real Gemini API (no stub), one turn per host at a time.
- [ ] Each host's lines are spoken by a distinct edge-tts voice (no API key, real audio).
- [ ] The session + final audio path are persisted in SQLite.

## What This Agent Does NOT Do (Out of Scope, v1)

- No transcript display, editing, or download (Phase 2).
- No 3+ host casts beyond the fixed persona set (Phase 3).
- No voice upgrades / custom/cloned voices (Phase 4).
- No sharing / public link / cloud hosting (Phase 5).
- No cross-session memory, topic history, or personalization (explicitly deferred).
- No mid-stream controls (pause/rewrite/branch) in v1.

## Key Constraints

- Single-user, single-tenant, local. No auth.
- Real Gemini key required (`:envvar:`AGENT_GEMINI_API_KEY`). No stubbed LLM on the tested path.
- TTS is edge-tts (free, no API key). Swappable later behind the same interface.
- Real-time delivery via Server-Sent Events (SSE); the browser `<audio>` element consumes the stream.
- SQLite for local state only; nothing leaves the machine except the Gemini/edge-tts network calls.
- KISS/DRY. The tested path must work first time.

## Phases of Development

### Phase 1 — Live two-host slice (THIS PHASE)

- **Goal:** The smallest first-time-right win — enter a topic, pick 2 hosts, press Generate, hear a
  coherent two-person conversation stream live, and download the mp3 when done.
- **Independent slices (parallel build units):**
  - `backend` (backend) — FastAPI app + SSE stream + dialogue generator (Gemini) + TTS (edge-tts) + SQLite persistence. deps: none.
  - `frontend` (frontend) — Next.js page: topic input, cast picker, Generate button, live `<audio>` player consuming the SSE stream, download link. deps: none (contracts with backend via `spec/api.md`).
  - `tests` (backend) — pytest suite (health + real generate producing streamed bytes + saved file). deps: none (exercises backend).
- **Key surfaces / files:**
  - backend: `src/main.py`, `src/config.py`, `src/db.py`, `src/graph/dialogue.py`, `src/graph/tts.py`, `src/graph/stream.py`, `src/schemas.py`, `src/prompts.py`.
  - frontend: `frontend/package.json`, `frontend/app/page.tsx`, `frontend/app/layout.tsx`, plus config.
  - tests: `tests/test_health.py`, `tests/test_generate.py`, `tests/conftest.py`.
- **Gate command:** `python -m pytest tests/ -q` (runs against REAL Gemini + edge-tts via repo-root `.env`).
- **How the user tests it (handoff seed):** Start backend (`uvicorn src.main:app`) and frontend (`cd frontend && npm run dev`); open the URL, type "future of remote work", select 2 hosts, click Generate. Expect a live two-voice conversation in the audio player; when done, a download link appears. The tested path is real (no stubs).

### Phase 2 — Transcript + richer control

- **Goal:** Show the live transcript alongside audio; allow download of transcript (txt/md). Wire the Phase-1 stub.
- **Independent slices:** `backend-transcript` (backend), `frontend-transcript` (frontend), `tests-transcript` (tests).
- **Gate command:** `python -m pytest tests/ -q` + frontend Playwright E2E.

### Phase 3 — 3-host casts + cast management

- **Goal:** Support 2–3 hosts; let the user name/edit fixed personas.
- **Independent slices:** `backend-cast` (backend), `frontend-cast` (frontend).

### Phase 4 — Voice upgrades

- **Goal:** Swap edge-tts for higher-quality / cloned voices behind the same TTS interface; per-host voice picker.

### Phase 5 — Sharing

- **Goal:** Public download link / shareable page for a finished podcast (single-user local still, but a share URL).
