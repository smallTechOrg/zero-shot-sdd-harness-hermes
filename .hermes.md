# Zero-Shot SDD Harness — Hermes project context

This repo is a **Hermes-native port** of the spec-driven-development harness for building
agents spec-first. It is the project context Hermes loads whenever the working directory is
inside this repo (`.hermes.md` is discovered by walking parents up to the git root).

## What this is

A lean, opinionated harness for building agentic software **spec-first**. One person with an
idea and one API key can drive a real, production-shaped agent into existence — and a senior
engineer opening the result finds a conventional, reviewable stack, not generated mush.

The six convictions (unchanged from the original):

1. **Spec is the source of truth.** Written before code; when spec and code disagree, the spec wins (`/zero-shot-sync`).
2. **Built for two audiences at once.** A non-coder drives it with a sentence; a senior engineer inherits a clean, reviewable stack.
3. **Lean harness, not a framework.** `harness/` is engineering mindfulness — kept small and Hermes-native.
4. **Smallest first-time-right win, phase by phase.** Each phase ships the smallest increment a human can test, and it must work the *first* time.
5. **A human gates every phase.** Autonomous within a phase; you test each increment before the next starts.
6. **Real LLM/API or it doesn't count.** Gates run against the real model with keys from `.env`.

## How Hermes uses this repo

- **Entry point:** `AGENTS.md` (root) — read first every session.
- **Rules:** `harness/rules/ai-agents.md`, `harness/rules/git.md`, `harness/rules/secret-hygiene.md`.
- **Patterns:** `harness/patterns/*` (spec-driven, phases, project-layout, tech-stack, code, test-driven, ui-ux, agentic-ai, engineering-practices).
- **Skills (entry points):** `harness/skills/zero-shot-build`, `…/zero-shot-fix`, `…/zero-shot-sync`.
- **Sub-agent roles:** `harness/agents/{agent-builder,spec-writer,code-generator,qa-auditor}.md` — invoked via Hermes `delegate_task`.
- **Spec:** `spec/` — the product spec (roadmap, architecture, capabilities, data, api, ui, agent).

## Invoking the skills (in a session with cwd inside this repo)

The three skills live as **repo artifacts** at `harness/skills/<name>/SKILL.md`. They are **not
installed Hermes skills**, so they do **not** auto-dispatch as native `/zero-shot-build` slash
commands unless copied into `~/.hermes/skills/`. The intended workflow: an agentic session reads
the SKILL.md (already in context via this harness) and follows its procedure.

- **Read + follow the file** (default, no install) — open `harness/skills/<name>/SKILL.md` and
  execute it.
- **Optional native slash command** — copy `harness/skills/<name>/` → `~/.hermes/skills/<name>/`
  for a typed `/zero-shot-build` via the registry. Convenience/portability only.

The three entry points:

- `zero-shot-build [idea]` — idea → working, verified, phased agent.
- `zero-shot-fix [target]` — diagnose + fix a bug / failing test / drift, then verify.
- `zero-shot-sync [scope]` — reconcile spec ↔ code (spec wins), then verify.

## Hermes runtime conventions

- Entry point is `AGENTS.md` (root) plus this `.hermes.md`; skills live at `harness/skills/<name>/SKILL.md`.
- Sub-agents (`harness/agents/*.md`) run via `delegate_task` (parallel, up to 3 children).
- User questions go through the **`clarify` tool (multiple-choice, single-select only — Hermes has NO multi-select)**; where a multi-select checklist would fit, ask more single-select questions instead. If `clarify` fails to load (e.g. Hermes on a plain command line), fall back to **plain-text questions asked one at a time** — never dump all questions in one message.
- Provenance: derived from `smallTechOrg/zero-shot-sdd-harness` (local copy at
  `/Users/sai/Workspace/Lab/zero-shot-sdd-harness`); provider-agnostic content (`spec/`,
  `harness/patterns/`, `harness/rules/secret-hygiene.md`) is shared, the runtime wiring here is
  Hermes-native.

## Hard rules (survive context compression)

- README must always be accurate — every command works exactly as written.
- **Never commit on `main` — no direct commits, ever.** All work (builds *and* harness/spec improvements) happens on a branch and lands via PR. Branch names carry a unique slug so they never clash with an existing local or remote branch (check `git branch -a`; if taken, add a differentiating suffix, e.g. `feature/<slug>-v0.1-2` or a date).
- **Questions: prefer `clarify` (multiple-choice); Hermes has NO multi-select.** Ask more single-select questions to cover what a checklist would — never guess. If `clarify` doesn't load, ask plain-text questions ONE AT A TIME, never all together.
- Never claim a test passed if you didn't run it. Show output.
- `main` is boilerplate-only — ABSOLUTELY. Nothing a `/zero-shot-build` run produces (application code, generated features, phase output) ever reaches `main`. App code lives on a feature branch and its PR targets the branch it was cut from (`--base $base`), **never `main`**. A build merged to `main` is a failed build — `git revert` it, don't rewrite history (see `harness/rules/git.md`).
- Commit + push are one atomic action: `git commit … && git push origin <branch>`.
- Never `git add -A` / `git add .` — stage specific files. `.env` is never staged.
- Real-key testing only. A stubbed pass is not a pass.
