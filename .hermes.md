# Zero-Shot SDD Harness — Hermes project context

This repo is the **Hermes-native** spec-driven-development harness for building agents
spec-first. Hermes loads this file whenever the working directory is inside this repo.

## What this is

A lean, opinionated harness: one person with an idea and one API key drives a real,
production-shaped agent into existence — and a senior engineer opening the result finds a
conventional, reviewable stack, not generated mush.

The six convictions:

1. **Spec is the source of truth.** Written before code; when they disagree, the spec wins (`/zero-shot-sync`).
2. **Built for two audiences at once.** A non-coder drives it with a sentence; a senior engineer inherits a clean, reviewable stack.
3. **Lean harness, not a framework.** `harness/` is engineering mindfulness — kept small, Hermes-native here.
4. **Smallest first-time-right win, phase by phase.** Each phase ships the smallest increment a human can test, and it must work the *first* time.
5. **A human gates every phase.** Autonomous within a phase; you test each increment before the next starts.
6. **Real LLM/API or it doesn't count.** Gates run against the real model with keys from `.env`.

## The Hermes execution model (the architecture, not a workaround)

- **The ROOT SESSION is the orchestrator.** It owns the human channel (`clarify`, with a
  plain-text one-question-at-a-time fallback), git/PR, and the server lifecycle. There is
  no delegated "agent-builder": on Hermes, workers can't spawn workers
  (`max_spawn_depth=1`), can't ask the user, and their background processes die on return.
- **Three dual-mode specialist roles** at `harness/agents/`: spec-writer, code-generator,
  qa-auditor. Delegate leaf work via `delegate_task` when available (parallel for
  independent slices); otherwise the root executes the role file **inline** as a checklist.
  The build never stalls waiting for a worker that cannot spawn.
- **Trust-but-verify:** a worker's "done" is advisory. On every handback the root checks
  the durable files exist and re-runs the gate cheaply; early returns at ~95% are normal
  and the root finishes the remainder inline.
- **The loop:** implement → run the REAL gate → read the actual output → fix → re-run.
  Real keys from `.env`; observed output only — never claimed.

## Invoking the skills — two ways in

The three entry points map to `harness/skills/{zero-shot-build,zero-shot-fix,zero-shot-sync}/SKILL.md`.
There are two ways to invoke them; **for either one, you (the agent) READ the matching
`SKILL.md` and follow it exactly, start to finish — the `SKILL.md` IS the process, never
improvise your own.**

**1. Plain words — works from a bare clone, zero setup.** When the user asks in natural
language to build / fix / sync an agent (e.g. "build me an agent that…"), immediately read
and follow the matching SKILL file:
- build an agent  → `harness/skills/zero-shot-build/SKILL.md`
- fix / debug     → `harness/skills/zero-shot-fix/SKILL.md`
- reconcile spec↔code → `harness/skills/zero-shot-sync/SKILL.md`

**2. The `/zero-shot-*` slash command — needs one-time registration.** Hermes rejects
*unregistered* slash commands at the CLI ("Unknown command") before the agent sees them, so
the literal `/zero-shot-build` only works once this clone is registered. One-time, one-line
edit to `~/.hermes/config.yaml` (tracks the repo — nothing to re-run after `git pull`):

```yaml
skills:
  external_dirs:
    - /absolute/path/to/this/clone/harness/skills
```

Restart Hermes after adding it. If a user reports "Unknown command: /zero-shot-build", point
them at this block — or tell them to just ask in plain words (path 1), which needs no setup.

## Hard rules (survive context compression)

- README must always be accurate — every command works exactly as written.
- Never claim a test passed if you didn't run it. Show output.
- `main` is boilerplate-only — ABSOLUTELY. Nothing a `/zero-shot-build` run produces ever
  reaches `main`. Builds branch from the current HEAD; the PR targets that base
  (`--base $base`), never `main`. A build merged to `main` is a failed build — `git revert`
  it (see `harness/rules/git.md`).
- Commit + push are one atomic action: `git commit … && git push origin <branch>`.
- Never `git add -A` / `git add .` — stage specific files. `.env` is never staged.
- Real-key testing only. A stubbed pass is not a pass; a present-but-dead key (401) is a
  BLOCKER naming the env var.
- Launch servers with the pinned interpreter (`.venv/bin/python -m src`) — never bare
  `python`. Only the root session serves; verify the URL live before presenting it.
- One batched LLM call per artifact — never a call per output line/token.
